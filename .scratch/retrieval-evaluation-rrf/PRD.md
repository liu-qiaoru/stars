# 检索评测与无权重 RRF 融合规格

Status: ready-for-agent（规格完整，可由 Agent 直接实施）

## 问题陈述

当前统一搜索把 SigLIP 视觉向量、Caption 文本向量和 PostgreSQL 全文检索结果合并为一个排序列表。不同模型的原始相似度分布不具备跨模型可比性，但现有 hybrid score 会直接使用归一化后的原始分数；单一向量来源的权重在除以总权重后还会抵消。因此 Caption 文本向量常以更高的原始 cosine 占据前列，但系统无法证明这是更高的真实相关性，还是不同模型分数尺度造成的排序偏差。

项目目前也缺少可复现的检索评测闭环。维护者无法系统回答以下问题：Caption 是生成事实错误、文本 embedding 语义过宽，还是融合策略过度奖励；SigLIP 是排序较低、被候选深度截断，还是根本没有召回目标；一次模型、提示词或排序修改究竟提升了哪些查询，又让哪些查询回退。

仅凭日常搜索体验或手工调整 Caption 权重会把症状掩盖为新的固定系数，无法定位根因，也无法在素材库、模型和索引变化后持续验证质量。

## 解决方案

建设一个仅供本地使用、由 PostgreSQL 持久化的检索评测功能，并在 Web 中提供内部评测页面。维护者可以创建并冻结版本化查询集，运行一次严格配置的多路召回，盲标图片文件或视频场景的相关性，诊断异常候选，并对同一候选快照计算现有 hybrid 排序与无权重 RRF 排序的指标。

首版 RRF 不直接替换普通搜索。它作为实验排序与现有算法并行计算，只有达到明确准入门槛后才可另行决定是否成为默认搜索排序。

无权重 RRF 不比较跨模型原始分数。每个独立检索信号依据来源内排名贡献 `1 / (60 + rank)`，所有有效信号的权重均为 1。首版信号包括 visual、caption 和 lexical：图片 visual 来自 image vectors；视频 visual 来自 video frame vectors，并在排名前按视频场景执行 MaxSim 折叠；caption 来自 caption text vectors；lexical 来自现有全文检索。video segment vectors 和查询扩展在首版评测中明确禁用。

评测以图片文件或视频场景为统一实体，避免 Caption asset、frame asset 和 transcript chunk 对同一内容重复计票。原始 collection、原始分数、来源内排名、RRF 贡献、模型版本、检索配置和语料规模摘要完整保留，用于定位问题而不是隐藏失败。

## 实施决策

### 模块边界

- 在 NestJS server 新增独立的评测领域，负责评测集、查询、运行编排、候选快照、标注、诊断、指标与导出。
- 复用现有查询向量化、Qdrant、PostgreSQL 全文检索、元数据补齐和场景 MaxSim，不另造媒体身份或 collection 配置规则。
- 仅为复用而抽取“最终 hybrid 排序前的多来源召回”边界；MVP 期间普通搜索行为不变。
- Next.js 新增仅供本地维护者使用的评测工作区；API 请求和响应沿用 snake_case。

### 评测集与查询生命周期

- 评测集拥有稳定身份和多个不可变版本；版本初始为草稿，可冻结。
- 冻结后禁止修改查询文本、类型、判断规则、目标身份及成员关系；修正必须创建新版本。
- 查询类型为“指定目标”或“自然发现”。每条查询保存文本、意图分类、必须满足、加分条件和明确排除。
- 指定目标额外保存图片文件，或视频文件与场景身份；冻结时目标必须存在且有效。
- 只有规则完整且已冻结的版本才能运行。首版建议约 40 条查询，约 40% 指定目标、60% 自然发现；系统只报告分布，不强制比例。

### 持久化模型

- 持久化评测集、版本、查询、运行、来源快照、候选、来源证据、排序输出、相关性判断、Caption 事实性判断和诊断注释。
- 统一使用 UUID 和数据库外键。
- 图片候选以文件为身份；视频候选以文件和场景为身份。没有可靠场景身份的时间窗口保持独立，禁止强行归并。
- 来源证据保存原始 collection 或 lexical 来源、原始分数、来源排名、贡献 asset、最佳帧时间和实际召回深度。
- 排序输出保存算法、最终排名、最终分数和逐信号贡献；当前 hybrid 与 RRF 使用相同候选行分别保存。
- 相关性判断绑定查询版本和语义候选，可在兼容运行间复用。
- Caption 事实性判断绑定准确的 Caption 内容、模型和 Prompt；重新生成后不得复用。
- 场景身份变化会使视频判断失效；删除的媒体保留历史引用并标记不可用。
- 数据库约束禁止同一查询版本、候选和判断类型出现重复有效记录。

### 运行状态与失败策略

- 运行状态依次为 `pending`、`retrieving`、`ready_for_labeling`、`labeled`、`reported` 或 `failed`，每次转换显式持久化。
- 运行快照保存评测集版本、素材库过滤、来源配置、召回深度、RRF 参数、模型身份、Caption Prompt、可检索图片数、视频场景数和 collection 点位数。
- 运行由维护者手动启动；新增素材不会自动修改或重跑评测集。
- 必需的 Embedding 服务、Qdrant 来源、数据库查询或元数据补齐失败时，运行进入 `failed` 并保存失败阶段；禁止省略来源继续生成指标。
- 孤儿 Qdrant 点、缺失场景边界和无效目标在评测模式下属于完整性错误，禁止静默跳过。
- 语料摘要不同的运行标记为不可直接比较；MVP 只记录和提醒，不复制媒体或 Qdrant collection。
- 新增素材后重新运行已有冻结查询集，新候选补充标注，兼容旧标注继续复用。

### 召回配置与候选深度

- 基线评测无条件关闭查询扩展和 video segment vectors。
- 请求 Caption 对比时，Caption 搜索或所需 Embedding 不可用必须直接失败。
- 图片 visual 使用 image vectors；视频 visual 使用 video frame vectors，并先按文件和场景执行既有 MaxSim，再分配来源排名。
- caption 使用 caption text vectors 及其模型感知的查询向量；lexical 使用现有 OCR 与 Transcript 全文检索路径。
- OCR 和 Transcript 保留不同诊断原因，但在 RRF 中共同构成一个 lexical 信号。
- 原始分数只决定各自来源内部排名，跨来源融合不得假设原始分数可比。
- 主标注池从每个启用来源最多取 top 20；深层诊断最多取语义候选总体的 20%，且每个来源不超过 50。
- 实际深度取配置深度与可用候选数的较小值，并随来源快照保存。
- 指定目标重点报告 top 5、10、20；只在深层或全库扫描中找到不算实际 top-K 成功。

### 语义候选对齐

- 图片的 visual、caption、OCR 证据按文件身份对齐。
- 视频 visual 与 caption 按文件和稳定场景身份对齐。
- Transcript 或 OCR 时间窗口只有满足单一、确定且文档化的时间重叠规则时才能归属场景；歧义或跨场景窗口保持独立。
- 实体对齐必须发生在计算 RRF 之前。
- 每个独立信号对同一候选最多贡献一次；信号内部多个 asset 或命中保留最佳排名，同时保留全部证据身份供诊断。
- 相邻窗口的展示合并不得改变排序身份或增加 RRF 票数；指标在任何展示性合并前计算。

### 无权重 RRF 基线

- 独立信号固定为 visual、caption、lexical，权重全部为 1，平滑常数固定为 60。
- 来源排名从 1 开始，单个信号贡献为 `1 / (60 + 来源排名)`；未命中不产生贡献。
- 最终 RRF 分数为各信号贡献之和，按分数降序；最终同分使用稳定的语义身份排序。
- 不增加额外的多信号奖励。主原因取贡献最大的信号，同贡献按固定优先级决定。
- RRF 分数只是排序值，不是概率或百分比，UI 不得显示成置信度。
- 最终结果同时保留原始 collection 分数、来源排名和逐信号贡献。

### 对照实验与标注

- 每次运行只召回和补齐一次，再基于同一份不可变来源证据分别计算当前 hybrid 与 RRF。
- MVP 不修改当前算法，报告分别展示两套排序，禁止混合分数；普通搜索仍返回当前生产排序。
- 初次标注隐藏来源、排名、原始分数、算法排名、Caption、Transcript 和 OCR。
- 候选按语义实体去重，并用持久化的稳定随机顺序展示，刷新不得重排。
- 图片展示原始媒体；视频播放准确场景或独立时间区间，而非仅展示最佳帧。
- 判断值为高度相关、部分相关、不相关、无法判断，指标分别映射为 2、1、0；无法判断排除出分母并单独统计。
- 标注增量保存并支持恢复；所有主池候选完成判断后运行才能进入 `labeled`。
- 保存初次判断后，或进入独立诊断视图后，才允许展示来源证据。

### 异常诊断

- 自动标记高排名但不相关的 Caption 结果，以及高度相关但 visual 排名过低或缺失的结果。
- 诊断视图展示来源证据、Caption 及来源信息、最佳视觉帧、场景区间、OCR/Transcript、原始分数、来源排名和两种算法排名。
- 根因分类固定为 Caption 事实错误、Caption 遗漏、文本 Embedding 语义过宽、视觉语义遗漏、抽帧遗漏、场景切分错误、实体对齐错误、融合排序错误和其他。
- 诊断只追加注释，不改写初次相关性判断；系统统计根因但不自动修改 Prompt、Embedding、权重或搜索配置。

### 指标、报告与晋级门槛

- 自然发现查询报告每种算法的 Precision@5、Precision@10、nDCG@10、nDCG@20。
- 指定目标查询报告 Hit@5、Hit@10、Hit@20、倒数排名和总体 MRR。
- 报告包含总体均值、逐查询指标、具备数据的查询分类切片和无法判断数量。
- 报告在数学上有效时给出 RRF 相对当前排序的绝对与相对变化，并列出提升和回退最大的五条查询。
- 跨运行比较必须显著显示语料摘要差异。
- RRF 只有同时满足以下条件才可提议成为默认排序：平均 nDCG@10 相对提升至少 5%；Precision@5 绝对下降不超过 0.02；改善查询多于回退查询；nDCG@10 下降超过 0.20 的严重回退不超过两条；Hit@10 和 MRR 不下降；核心查询类别无明显系统性回退。
- 晋级前必须人工复核提升和回退最大的五条查询。
- MVP 不通过调参强行过线；只比较预先确定的当前算法和无权重 RRF。未来调整 k 或权重前必须扩充查询集并建立 development/holdout 分区。

### API 与 Web 行为

- API 支持创建和列出评测集、创建草稿版本、维护草稿查询、冻结版本及查看查询分布。
- API 支持从冻结版本和所选素材库启动固定基线运行，查看运行状态、来源/语料快照、标注进度、下一个未标候选和媒体元数据。
- 保存相关性判断的操作必须幂等；支持列出异常候选、保存诊断、完成标注、计算报告、比较兼容运行和导出数据。
- 主标注未完成时拒绝生成报告。校验错误返回 400，资源不存在返回 404，不可变版本冲突返回 409，依赖或完整性失败返回带运行阶段证据的明确错误。
- 响应不得泄露向量、API Key 或无关本地文件信息。
- Web 新增内部评测入口，包含查询集、运行状态、盲标、报告和诊断视图。
- 指定目标选择支持图片或精确视频场景，并隐藏模型文本。
- 盲标支持键盘操作，明确展示四种判断。报告不得把 RRF 分数表示为概率。
- 失败运行和部分索引问题持续可见；重试必须创建新运行并保留失败历史。

### 文档与可观测性

- 更新 API 契约、架构文档和向量检索文档，明确评测领域、生产排序与实验 RRF 的区别，以及 raw cosine 只在来源内部有意义。
- 实施导致运行拓扑、Schema 或搜索行为变化时同步更新 Living Documentation。
- 结构化日志记录运行身份、查询数、启用来源、实际深度、候选数、去重数、耗时、失败阶段和指标计算耗时。
- 禁止记录查询向量、API Key、完整本地媒体内容和盲标阶段隐藏的证据。

## 测试决策

- 主要测试边界是公开的评测运行行为：给定冻结评测集、PGlite 媒体和标注事实、mock 向量来源，执行完整运行并验证来源快照、语义候选对齐、两种排序、标注完成、报告指标与失败状态。
- 优先通过评测 service/controller 边界断言外部行为，不测试私有编排方法或 SQL 形状。
- 复用现有 PGlite 模式测试 Schema 约束、repository 行为、标注复用、不可变冲突、删除历史和状态转换。
- 复用现有 mock Qdrant 与搜索测试模式覆盖来源结果、孤儿点、服务失败、Caption 开关和媒体过滤。
- 复用场景 MaxSim 测试，并新增高层用例证明帧命中先折叠再分配来源排名，单个场景只有一次 visual 贡献。
- 为 RRF 增加纯函数测试，覆盖排名计算、来源缺失、同分、单信号单票和稳定 tie-break。
- 为 Precision@K、nDCG@K、Hit@K、MRR 增加纯函数测试，覆盖空结果、无法判断、少于 K 和分级相关性。
- API 测试覆盖草稿/冻结生命周期、指定目标完整性、规则不完整、标注未完成、幂等更新、不兼容复用和导出。
- Fail-fast 测试证明 visual、caption、lexical、元数据补齐或场景边界失败会使运行失败，且绝不生成成功报告。
- 测试证明基线评测不会调用查询扩展和 video segment vectors，即使普通应用配置启用了它们。
- 测试证明当前 hybrid 和 RRF 使用完全相同的持久化候选来源快照。
- 测试证明语料摘要不同会提示不可直接比较，而不是静默视为等价。
- Web 测试覆盖查询编写、冻结、盲标隐藏、增量保存、恢复、视频场景播放、失败展示、诊断揭示和报告渲染。
- API client 测试覆盖所有新增路由和 snake_case 契约。
- 测试不得依赖真实 PostgreSQL、Qdrant、Ollama、SigLIP、MiniLM 或下载本地模型。
- 不测试私有 helper、ORM 实现细节、CSS 类名或无关文案；表达失败原因和盲标保证的文案除外。
- 完整验收运行 shared、server、web 的 check，并确保数据库 Schema 与迁移产物一致。

## 非目标

- MVP 阶段修改默认生产搜索排序。
- 自动调整 RRF k、信号权重、分数校准或阈值。
- 训练 Learning-to-Rank、分数校准器、Cross-Encoder 或 Embedding 模型。
- 自动改写 Caption Prompt、重新生成 Caption 或更换 VLM。
- 在首轮基线中启用查询扩展或 video segment vectors。
- 将代表同一语义信号的每个 collection 都视为独立 RRF 票。
- 为每次运行复制不可变媒体、PostgreSQL 事实或 Qdrant collection。
- 素材库变化时自动重跑评测。
- 多用户认证、标注者角色、标注一致性或仲裁流程。
- 线上 A/B 测试或终端用户行为埋点。
- 使用 LLM 生成完整基准查询集。
- 穷举标注整个媒体库，或声称联合候选标注等于完整召回真值。
- 将 CSV/JSON 导入作为 PostgreSQL 评测状态的权威替代。
- 通用实验平台或任意排序表达式构建器。

## 补充说明

- 第一版基准目标约为 40 条人工查询，但实施验收不依赖全部标注完成。功能只需能用代表性小样本完成创建、冻结、运行、标注和报告，并可供维护者继续填充。
- 小素材库中固定 top 300 会误导判断；动态深度、top-K 成功和语料规模必须同时展示。
- 联合候选标注无法证明不存在未召回的相关内容；指定目标、深层诊断和未来加入 hard negative 用于缓解这一限制，但不得假装拥有穷举召回真值。
- 高 Caption cosine 只是在 Caption Embedding 来源内部的证据，不证明它比低 SigLIP cosine 更相关。
- Caption 错误占主导时，必须区分 VLM 事实错误与“Caption 正确但文本 Embedding 过宽”；只有证据指向融合层时才调整融合权重。
- 指定目标未进入 visual top K 时，应先检查深层 visual 排名、最佳索引帧、场景边界和索引完整性，再判断 SigLIP 是否不适用。
- 本地 Markdown tracker 中的本规格是后续拆票来源。对已确认范围的重大修改必须创建新规格版本，不得在实施中静默漂移。
